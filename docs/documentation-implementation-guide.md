# Documentation

## When to create a new doc

Check if a canonical doc already exists — update it instead of creating a new one. Create a new doc only when introducing new process, decisions, or operational steps.

## Writing rules

- Document implemented behavior, not aspirational design.
- Keep one source of truth. Link instead of duplicating.
- Use explicit language: `must` and `must not` for requirements.
- State ownership, boundaries, and decision rules (`if X, do Y`) directly.
- Keep examples aligned with the current codebase.
- Avoid vague wording (`might`, `usually`, `etc.`).
- Be concise. Prefer bullets and short sections.

## What makes a good doc

A contributor must be able to execute the process without guessing. Commands, paths, and names must be current. Validation steps must be testable. Related docs should be linked, not duplicated.
